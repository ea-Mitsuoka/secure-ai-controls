# 個人目標:社内Gemini Enterprise の統制インフラをIaC自動構築し、監査ログ監視基盤を検証環境に実装(クライアントゼロ)

対象期間:2026年4月〜2027年3月 / テーマ:AIタスクフォースでの安全なAI環境構築(統制インフラIaC＋監査ログ)

______________________________________________________________________

## 目標項目(何を)

社内Gemini Enterpriseの「統制されたインフラ」をTerraformでIaC自動構築する。起点3点(①機密データフィルタリング ②プロンプト監査ログ監視 ③生成物の法的適合)の統制要件をGoogle Cloud機能へマッピングし、検証環境で動作する形に実装する。

## 達成基準(D〜S)

| 段階 | 達成基準(状態＋数値) |
| --- | --- |
| **S(120%〜)** | Aに加え(いずれも体制・承認に依存):分科会レビューを通過し社内Gemini本番環境の一部に適用、または統制インフラIaCが全社標準構成として採用、または外販パッケージの技術核として位置づけ。かつスペシャライゼーション／事例資料として提出済。 |
| **A(110%)** | Bに加え:本人が実施したレッドチーミング(プロンプトインジェクション・機密データ持出の試行)でガードレールの検知動作を実証し結果レポートを作成。加えて検知ルール／統制項目を検証環境で拡充(例:異常なツール連鎖・決定反転の検知を追加)。 |
| **B(100%・基準)** | 2027/3末までに次の3点を作成し、検証環境で「機密データ検知」と「プロンプト監査ログ取得→アラート」の動作を実証。(1)起点3点の統制要件をGoogle Cloud機能(DLP API／VPC-SC／IAM／Cloud Audit Logs／Model Armor 等)へマッピングした設計書、(2)検証環境で構築できる統制インフラのTerraform一式、(3)監査ログ収集→検知ルール→アラートのパイプライン。成果物=設計書＋IaCコード＋検知ルール＋実証ログを納品済。 |
| **C(〜80%)** | 設計書は完成したが、IaCまたは監査ログ基盤の一部が検証環境で未動作、もしくは実装に他者サポートを要した。 |
| **D(50%以下)** | 設計・調査までで、動作する成果物(IaCコード・監査ログ基盤)がない。 |

**測定方法**:実装した統制項目数(起点3点→GCP機能のマッピング実装数)、検証環境での検知動作の成否(機密データ検知・監査ログ取得)、レッドチーミング結果(検知できた攻撃パターン数)、拡充した検知ルール数。全社承認・採用・本番適用は裁量外のためSの上振れ要素に限定し、B・Aの判定には用いない。

## 達成過程(何を・どの順で・何を準備するか)

| 段階(時期) | やること(活動・準備) | この段階で見極める／確定すること |
| --- | --- | --- |
| **① 要件整理・方式比較(7〜8月)** | 起点3点(機密データフィルタ／監査ログ監視／生成物の法的適合)を統制要件に分解し、GCPのどの機能(DLP API／VPC-SC／IAM／Cloud Audit Logs／Model Armor 等)で満たすか候補を比較検証。分科会の関係者と担当境界・前提を確認。 | 各統制要件の実装方式。技術主幹との担当境界。 |
| **② 基盤試作(9〜10月)** | 検証環境に統制インフラの骨格をTerraformで試作(プロジェクト構成／IAM最小権限／VPC-SC)。機密データ検知と出力フィルタを組み込んで動かす。 | 統制インフラの再現性。検知の当たり方。 |
| **③ 監視・検知の実装検証(11〜12月)** | 監査ログの収集→検知ルール→アラートを実装し、検証環境で機密データ検知とログ→アラートを通しで実証。統制項目をモジュール化し差し替えに耐える構造に整える。 | パイプラインの動作。統制項目変更への耐性。 |
| **④ 攻撃実証・レビュー(1〜3月)** | 本人でレッドチーミング(プロンプトインジェクション／機密持出の試行)を実施し検知動作を確認、検知ルールを増やす。分科会レビューにかけ、(上振れ)本番一部適用を上長と調整。 | 検知の実効性。レビュー指摘と本番適用可否(裁量外)。 |

> **期中評価(10月)の中間成果**:②基盤試作段階の「検証環境で立つ統制インフラの骨格＋機密データ検知の初動」を中間成果として示す。

## 想定される課題と対策

- **体制・承認経路が未確定で統制ルールが全社承認されない**:→ 基準(B)を「検証環境で動作する統制インフラ・設計・実証成果物の完成」に置き、全社承認・本番採用はA/Sの上振れに限定。
- **起点3点そのものが期中に変わる**:→ 統制項目をTerraformモジュール化し、項目差し替えに耐える構造で作る(設計変更を前提)。
- **技術判断が他メンバー(技術主幹・インフラ構築)と重なる**:→ 「統制をIaCで実装する」部分を内部統制主担当の範囲として設計書に明記し、レビュー協働の形で境界を定義(Secure by Design=統制の設計者が実装まで一貫して持つ)。
- **本番権限が下りない**:→ 検証環境(クライアントゼロ)でB完結。本番適用はA以上の上振れに限定。
